RapidIdentity Root Filesystem Expansion

Use this procedure to add approved capacity to the root filesystem of a CentOS-style RapidIdentity appliance. It covers an XFS root filesystem on LVM, using either free space already present in the root volume group or new capacity added to the appliance's existing virtual disk.

For an on-premises appliance, the customer or infrastructure owner confirms the recovery point and changes the virtual disk; Support performs the approved technician-mode partition and LVM work. Confirm the maintenance window, console access, validated backup or snapshot, and rollback owner before changing storage. Route hosted or cloud infrastructure changes through DevOps.
Device names in this guide are placeholders. Do not assume that the disk is /dev/sda, the new partition is /dev/sda4, the volume group is centos, or the root logical volume is /dev/centos/root.

1. Decide whether expansion is the right action

  1. Confirm which appliance is full and record its role, such as DB, DSS/Connect, Portal/ARMS, or another module-specific node.
  2. Confirm that / is the filesystem that needs capacity, then identify the largest top-level path.
df -h /
du -x -h --max-depth=1 / | sort -h
  1. Drill into the largest path until the source is understood. Accumulated or failed database backups, RapidIdentity logs, Connect logs, and database or audit growth are common possibilities, but none should be assumed without checking.
  2. If the root filesystem is already full and PostgreSQL or RapidIdentity will not start, first free only enough known-safe space for the appliance to operate. Typical candidates are confirmed expired or failed backup archives and the oldest rotated or compressed logs.
  3. Do not delete PostgreSQL data files, WAL files, arbitrary files under /var/lib/pgsql*, or database tables to make room.
  4. Continue with expansion when cleanup would provide only temporary relief or the approved capacity change requires a larger filesystem.

2. Record the current appliance and storage layout

  1. Confirm that you are connected to the intended appliance.
hostname
ip addr
  1. Capture the current filesystem, disk, LVM, and root-mount details.
df -h
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
pvs
vgs
lvs
findmnt -no SOURCE,FSTYPE /
  1. Write down the actual disk, new partition path, volume group, root logical volume, root mount point, and filesystem type that the outputs identify.
  2. Confirm that the root filesystem is on LVM and that its filesystem type is xfs. If either condition is false, stop and use the procedure approved for that layout.
  3. If the appliance hosts the local database, confirm that a recent independent database backup and a VM snapshot or equivalent rollback point exist.
  4. Check the root volume group's free space in vgs. If it already contains enough approved free space, do not modify the disk, partition table, physical volumes, or volume group. Skip to section 6 and extend the root logical volume.
  5. If the root volume group does not contain enough free space, confirm that the infrastructure owner has enlarged the correct existing virtual disk by the approved amount before continuing.

3. Make the added capacity visible to Linux

  1. Run lsblk and compare the device size with the approved infrastructure change.
  2. If an expanded SCSI disk does not show its new size, rescan that specific device. Replace sdX with the disk confirmed in the previous step.
echo 1 > /sys/class/block/sdX/device/rescan
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
  1. If the new size still is not visible, stop. Coordinate an approved reboot or have the infrastructure owner correct the virtual-disk change.
  2. Inspect the correct disk and its partition table before making a change.
fdisk -l /dev/sdX
The partition steps below describe the classic DOS/MBR layout used by older CentOS-style appliances. If the disk uses GPT, has no free partition entry, exceeds the supported MBR range, or has a nonstandard layout, stop and use the current approved partitioning method for that system.

4. Create a new LVM partition

  1. Open the disk confirmed in the previous section. The new partition must use only the capacity added to that disk. Do not delete or recreate an existing partition.
fdisk /dev/sdX
  1. Create the next available primary partition, accept the approved start and end sectors, set the new partition type to Linux LVM, and print the proposed table before writing it.
n
p
<next-free-partition-number>
<enter for the default first sector>
<enter for the default last sector>
t
<new-partition-number>
8e
p
w
  1. Ask the kernel to reread the partition table, then confirm that the new partition appears at the expected size.
partprobe /dev/sdX
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT

If the partition does not appear, has the wrong size, or shows an unexpected filesystem or mount, stop before running pvcreate. A maintenance reboot may be required before Linux can use the new partition.

5. Add the new partition to the volume group

  1. Create an LVM physical volume on the new partition only, then verify it. Replace /dev/sdXN with the actual partition path.
pvcreate /dev/sdXN
pvs
  1. Add that physical volume to the volume group that contains the root logical volume.
vgextend <vg-name> /dev/sdXN
vgs

Continue only when vgs shows the expected additional free space in the root volume group.

6. Extend the root logical volume

  1. Confirm the root logical-volume path again before extending it.
lvs -o lv_path,vg_name,lv_name,lv_size
findmnt -no SOURCE /
  1. Extend the root logical volume by the approved amount.
lvextend --size +<approved-size> /dev/<vg-name>/<root-lv-name>

If the approved plan is to assign every free extent in the volume group to the root logical volume, use:

lvextend --extents +100%FREE /dev/<vg-name>/<root-lv-name>
Use +100%FREE only when consuming all free volume-group space is the approved result. Reserve free extents when another logical volume or future change needs them.
  1. Confirm that the logical volume grew and that the volume group's remaining free space matches the plan.
lvs
vgs

7. Grow the XFS filesystem

  1. Reconfirm that / is the intended mount and that its filesystem type is xfs.
findmnt -no SOURCE,FSTYPE /
  1. Grow the mounted XFS filesystem, then verify the new usable size.
xfs_growfs /
df -h /

Pass the mounted filesystem path to xfs_growfs. If the filesystem is ext4 or another type, do not run this command; use the approved procedure for that filesystem.

8. Restore service and validate the appliance

  1. Capture the final storage state and review the system log for disk, LVM, or XFS warnings.
df -h /
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
pvs
vgs
lvs
tail -n 200 /var/log/messages
  1. If PostgreSQL stopped because the appliance was full, start or restart it through the appliance's Local Database menu. Do not use install or uninstall as a service-recovery action.
  2. Start or validate RapidIdentity and confirm that the expected UI is available.
  3. For a multi-appliance environment that was stopped, bring the database node up before the non-database nodes, following the environment's approved startup order.
  4. Confirm login, database connectivity, jobs, and the appliance role's normal functions.
  5. Do not reboot only to prove that the LVM and XFS changes persist. If the approved change plan requires a reboot for another reason, repeat df -h /, lsblk, vgs, and lvs afterward.
  6. Attach the before-and-after outputs and the exact disk, partition, PV, VG, LV, and filesystem commands to the change or support record.

9. Prevent the filesystem from filling again

  1. Identify the source of the growth instead of treating expansion as the complete fix.
  2. If database backups consumed the space, review the daily cleanup job, failed backup artifacts, and the approved retention period. Do not assume that creating scheduled backups also removes old archives.
  3. If RapidIdentity or Connect logs consumed the space, confirm log rotation and check whether debug or trace logging was left enabled.
  4. If database or audit data consumed the space, use the product-supported retention or cleanup path. Do not truncate tables as routine disk recovery.
  5. Review Security > Retention Policies. Change an Always Retain setting only when the data does not require long-term preservation and the customer, security, audit, and legal boundaries allow the change.
  6. Monitor the filesystem and the next scheduled backup or retention cycle after recovery. If usage rises quickly, stop and investigate the continuing source before adding more capacity.

When to stop

  • The disk shown by Linux does not match the infrastructure change.
  • The partition table is GPT, full, damaged, or different from the expected layout.
  • The proposed partition overlaps existing storage or the target partition is already in use.
  • The root filesystem is not on LVM or is not XFS.
  • The volume group or root logical volume cannot be identified unambiguously.
  • Any command reports an unexpected device, size, filesystem, or allocation.
